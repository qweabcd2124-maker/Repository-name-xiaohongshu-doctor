# 小红薯医生 — 演讲与展示策略

## 一、展评演讲结构（3 分钟版）

### 黄金公式：Hook → 体验 → 数据 → 收尾

```
[0:00-0:10] Hook
  评委走到展位 3 秒内开口：
  "发小红书最痛苦的是什么？不是写笔记，是发完没人看还不知道为什么。"

[0:10-0:30] 启动演示
  "来，我给你看——"
  → Ctrl+V 粘贴一张截图，填标题，点诊断
  "30 秒，5 个 AI Agent 同时开始分析。"

[0:30-1:30] 等待时讲架构（60 秒填充）
  "传统工具就是一个 AI 给你建议——拍脑袋。
   我们不一样。5 个 Agent 各看一个维度，然后互相辩论。
   内容分析师可能说标题不错，但用户模拟器会说
   '目标用户其实不关心这个话题'——这种冲突本身就是价值。

   而且，我们不是跟 AI 的感觉对比，
   是跟真实数据对比——我们分析了 500 多条真实笔记，
   知道每个品类的爆款标题平均多少字、最佳发布时间是几点。"

[1:30-2:30] 报告出来了
  → 展示分数、雷达图
  "看，72 分。标题偏短——美食爆款平均 18 字，这篇只有 8 个。
   封面饱和度低于平均——美食类高饱和封面点击率高 28%。
   这些数字不是编的，是从数据里算出来的。"
  → 快速展示模拟评论
  "这是 AI 预测的评论区——有人种草、有人质疑、有人求链接。"

[2:30-3:00] 收尾
  "我们还能一键生成优化版标题、导出诊断卡片。
   有什么想深入看的吗？"
```

### 节奏控制

- 如果诊断提前出结果 → 跳过架构讲解，直接展示报告
- 如果评委中途打断提问 → 好事，说明感兴趣，跟着他的问题走
- 如果评委赶时间 → 30 秒版：一句话 + 直接翻开预跑好的报告截图

### Demo 时间管理

Demo 是最危险的环节（网络、API 都可能出问题）。

**Plan A**：现场 Live Demo（最加分）
- 提前打开页面 + 剪贴板备好截图
- 如果 15 秒内没出结果 → 说"网络原因，我们切到预录版本"

**Plan B**：预录视频（最安全）
- 2 分钟精剪版，按 video_script.md 拍
- 视频播放时旁白补充技术细节

**Plan C**：截图走读（兜底）
- 准备 4 张高清截图（首页 → 诊断中 → 报告 → 辩论时间线）
- 一页页翻，配合讲解

## 三、Q&A 应对策略（5 分钟）

### 高频问题预判

| 问题 | 30 秒回答 | 陷阱提醒 |
|------|----------|---------|
| "跟灰豚/新红什么区别？" | "他们做数据监控——告诉你结果是什么。我们做诊断——告诉你为什么，以及怎么改。完全不同的产品定位。" | 不要贬低竞品 |
| "多 Agent 不就是多调几次 API？" | "每个 Agent 有独立视角和 prompt。关键是辩论轮——Agent A 会引用 Agent B 的具体观点进行反驳。这跟串行调 5 次 API 是本质不同的。" | 可以打开辩论时间线展示 |
| "数据从哪来的？" | "公开笔记的结构化特征——标题长度、标签数量、互动数据。我们不存储原文，不涉及用户隐私。" | 如果用了爬虫别直说 |
| "怎么证明诊断是准的？" | "两个方法：一是用已知爆款做反向验证，小红薯医生评分与真实互动数据正相关；二是基线数据来自真实笔记统计，有统计学检验支撑。" | 准备一个具体数字 |
| "这能赚钱吗？" | "SaaS 订阅。免费版每月 3 次诊断引流，Pro ¥29/月无限用。MCN 版按席位。小红书 8000 万创作者，1% 转化就是 80 万付费用户。" | 不要说"还没想过" |
| "你们才 13 岁，能维护这个项目吗？" | "代码全在 GitHub，架构模块化。而且我们自己每天都用小红书，是自己的刚需——这保证了我们不会放弃。" | 自信但不自大 |

### 回答原则

1. **30-60 秒内回答**。超过 60 秒 = 你不清楚重点
2. **不知道就说不知道** + "但我们的思路是..."。诚实 >> 装懂
3. **回答完反问一句** "这回答了您的问题吗？" → 掌控节奏
4. **如果多个评委同时问** → 选最技术的那个先回答（技术评委占比高）

## 四、身份策略

### "全场最年轻"怎么用

```
❌ 错误用法："我们才 13 岁，请大家多多包涵"
   → 要同情分，减分

❌ 错误用法：不提年龄，假装成年人
   → 浪费了天然差异化

✅ 正确用法：先展示实力，让评委自己发现
   → 在队伍介绍材料里写 "平均年龄 13 岁"
   → 产品演示结束后，如果评委惊讶，自然回应：
     "对，我们四个都是初中生。但我们每天都在发小红书——
      这不是别人的需求，是我们自己的问题。"
```

### 分工建议

| 角色 | 人选建议 | 场景 |
|------|---------|------|
| 主讲人（展评） | 表达最流利的 | 展台 3 分钟讲解 |
| Demo 操作 | 最熟悉产品的 | 展评 + 决赛操作电脑 |
| 技术回答 | 写代码最多的 | Q&A 技术追问 |
| 社交 / 投票 | 社交能力强的 | 巡展时去其他展位交流 |

决赛允许多人上台，建议 2 人：一人讲 + 一人操作 Demo。

## 五、展评当天时间分配

```
10:00 - 10:30  热身准备，确认设备、网络、备用方案
10:30 - 12:00  上午展评高峰（评委精力最好）
               → 全力应对每一位评委，拿出最佳状态
12:00 - 13:00  午间低峰
               → 轮换休息、吃东西
               → 1 人留守展位应对散客
13:00 - 15:00  下午展评
               → 状态可能下降，准备提神
               → 评委也累了，更需要 "wow moment" 打破疲劳
15:00 - 15:30  收尾
               → 最后冲刺，可能有评委补看
```
