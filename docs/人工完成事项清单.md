# 薯医 NoteRx — 需要人工完成的事项

本文档列出 **无法通过代码自动化完成**、需要团队成员人工参与的任务。
请根据黑客松比赛最新要求和团队分工进行调整。

---

## 1. API Key 配置

| 项目 | 详情 |
|------|------|
| **做什么** | 在 `backend/.env` 中填写真实的 API Key |
| **涉及字段** | `OPENAI_API_KEY`（必须）、`ANTHROPIC_API_KEY`（可选）、`LLM_PROVIDER`（默认 openai） |
| **模板** | 复制 `.env.example` 到 `backend/.env`，按注释填写即可 |
| **风险** | 没有 Key 则后端所有 AI 功能不可用，前端自动回退到 fallback 演示数据 |

---

## 2. 真实数据采集与清洗（P0 / 赛前）

| 项目 | 详情 |
|------|------|
| **目标** | 每垂类 2k–3k 条真实小红书笔记数据，建立可信 baseline |
| **当前状态** | `scripts/seed_data.py` 为合成数据（6 个垂类各 500 条），可用于演示但答辩说服力不足 |
| **要做的** | ① 获取合规数据源（手动导出 / 授权爬虫 / 第三方数据平台）；② 清洗去重；③ 写入 `notes` 表；④ 重跑 `python scripts/compute_baseline.py` |
| **注意** | 仅使用可获得授权或公开范围的样本；答辩准备「合规说明」一句话；README 写明数据来源和边界 |
| **产出** | 更新后的 `baseline.db`（不提交到 Git，通过 `make data` 可复现） |

## 4. 可选增强（如时间允许）

### 4.1 CLIP 视觉语义集成

- 选择 `open_clip` 本地模型或云端 embedding API
- 集成到 `backend/app/analysis/image_analyzer.py`
- 需要评估模型体积对部署的影响

### 4.2 本地 OCR 备选（PaddleOCR）

- 在 `backend/app/analysis/ocr_processor.py` 中添加本地 OCR 分支
- 仅在 API 不可用或成本受限时启用
- 需要安装 PaddleOCR 依赖（体积较大）

### 4.3 Token 与成本优化

- 根据后端日志中的 token 统计（orchestrator 已自动记录），决定是否：
  - 将非裁判 Agent 降级为 `gpt-4o-mini`
  - 减少辩论轮为 2 Agent 对抗 + 裁判
- 这些是架构决策，需要人工判断
