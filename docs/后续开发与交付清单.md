## 1. 文档说明与当前基线

### 1.1 当前已实现（基线）

- **前端**：React + Vite + Tailwind；首页（文字/图片上传）、诊断动画页、报告页（雷达图、Agent 折叠、模拟评论、卡片导出）；API 失败时使用 `frontend/src/utils/fallback.ts`。
- **后端**：FastAPI；`POST /api/diagnose`（multipart）、`POST /api/upload-image`、`GET /api/baseline/{category}`；多 Agent 编排（并行第一轮 + 辩论 + 裁判）；文本分析（jieba）、图像分析（OpenCV/启发式）、Baseline 对比（SQLite）。
- **数据**：`scripts/init_db.py`、`scripts/seed_data.py`、`scripts/compute_baseline.py`；`backend/data/baseline.db` 由 `.gitignore` 忽略（本地生成）。

### 1.2 本文档定位

- **不是**对已实现功能的重复说明；**是**对白皮书 [`noterx_whitepaper.md`](../noterx_whitepaper.md)、产品与路演叙事的 **差距清单** 与 **可执行任务书**。
- 每项尽量包含：**目标**、**现状**、**要做事项**、**验收标准**、**主要改动文件/模块**、**风险与备注**。

## 2. 产品与体验缺口

### 2.1 笔记上传：小红书链接导入（P0）

| 项目 | 内容 |
|------|------|
| **目标** | 用户粘贴笔记分享链接，系统自动拉取或可解析出标题、正文、标签、封面 URL（在合规前提下）。 |
| **现状** | 仅支持手动标题/正文/标签 + 封面文件上传；无链接输入。 |
| **要做事项** | ① 前端 `Home.tsx` 增加「链接」Tab 或输入框；② 后端新增接口或扩展现有诊断流：`POST /api/parse-link` 或诊断前一步解析；③ 定义支持的 URL 形态（短链、标准笔记页）；④ 失败策略：提示用户改用截图 OCR 或手动粘贴。 |
| **验收标准** | 至少一种官方/常见分享格式在本地可稳定走完「链接 → 预填表单 → 一键诊断」；失败时有明确文案，不白屏。 |
| **主要文件** | `frontend/src/pages/Home.tsx`、`frontend/src/utils/api.ts`；`backend/app/api/` 新模块或 `diagnose.py`；可选 `backend/app/utils/link_parser.py`。 |
| **风险** | 平台反爬、协议与黑客松规则；**不得**存储用户隐私或非公开数据 unless 允许；答辩时准备「合规说明」一句话。 |

### 2.2 截图 OCR 接入主流程（P0）

| 项目 | 内容 |
|------|------|
| **目标** | 用户仅上传小红书截图时，系统自动抽取标题、正文、标签（及可见信息），减少手工输入。 |
| **现状** | `backend/app/analysis/ocr_processor.py` 具备基于视觉 LLM 的思路，**主诊断路径未调用**；用户仍需手填标题。 |
| **要做事项** | ① 在 `Orchestrator.run` 或 `diagnose` 入口：若 `cover_image` 存在且 `title/content` 为空或标记为「来自截图」，先调 OCR；② 合并/覆盖策略（OCR 结果与用户手改冲突时以谁为准）；③ 成本控制：可选「仅 OCR 不跑全量 Agent」预览模式。 |
| **验收标准** | 纯截图路径：最小可用为「OCR 出标题 + 部分正文」即可上线 demo；日志中可追踪 OCR 是否成功。 |
| **主要文件** | `backend/app/api/diagnose.py`、`backend/app/agents/orchestrator.py`、`backend/app/analysis/ocr_processor.py`。 |
| **风险** | API 费用、延迟；需 fallback（当前前端已有整页 fallback，后端 OCR 失败应降级为提示用户补全）。 |

### 2.3 免责声明与用户预期（P0）

| 项目 | 内容 |
|------|------|
| **目标** | 降低「被质疑为算命」风险，符合白皮书风险表「AI 诊断仅供参考」。 |
| **现状** | README 无固定免责；报告页无持久提示。 |
| **要做事项** | ① `Report.tsx` 页脚固定一行免责声明；② `README.md` 与产品内「关于」简短说明；③ 若导出卡片含分数，卡片底部附一句免责更小字。 |
| **验收标准** | 任何首次进入报告的用户可见免责；对外材料（Slide）可引用同一句。 |
| **主要文件** | `frontend/src/pages/Report.tsx`、`frontend/src/components/DiagnoseCard.tsx`、`README.md`。 |

### 2.4 诊断进度与真实阶段同步（P1）

| 项目 | 内容 |
|------|------|
| **目标** | 动画/进度与后端实际阶段一致，避免「已跳到最后一步但接口仍在等 LLM」。 |
| **现状** | `Diagnosing.tsx` 用固定间隔推进步骤，与 API 返回时间解耦。 |
| **要做事项** | **轻量**：根据 axios 超时调大步骤间隔或在收到响应后立即收尾；**标准**：SSE/WebSocket 或轮询 `task_id` 多阶段状态；后端 `Orchestrator` 分段 yield 状态。 |
| **验收标准** | 网络慢时不出现「步骤全绿但长时间不跳转」；或步骤与后端日志阶段可对应。 |
| **主要文件** | `frontend/src/pages/Diagnosing.tsx`；`backend/app/api/diagnose.py`（若加流式）。 |

### 2.5 一键优化（P2，白皮书 P2）

| 项目 | 内容 |
|------|------|
| **目标** | AI 重写标题、重排正文、封面方向建议；与「可选」一键生成优化版笔记一致。 |
| **现状** | 裁判 Agent 可能返回 `optimized_title`；无完整「优化正文」「封面方向」结构化产品能力。 |
| **要做事项** | ① JSON schema 扩展：`optimized_content`、`cover_direction`（构图/色系/文案区占比建议）；② 前端「复制」「应用草稿」按钮；③ 可选单独按钮触发第二请求以省首包 token。 |
| **验收标准** | 答辩可演示「原文 → 优化标题 + 一段改写正文 + 封面 checklist」。 |
| **主要文件** | `backend/app/agents/prompts/judge_agent.py`、`orchestrator.py`；`frontend/src/pages/Report.tsx`。 |

## 3. 数据与 Baseline 护城河

### 3.1 真实数据采集与清洗（P0 / 赛前）

| 项目 | 内容 |
|------|------|
| **目标** | 白皮书所述：每垂类 **2k–3k** 条真实笔记字段（标题、封面特征或 URL、标签、互动、时间、粉丝档位等），建立可信 baseline。 |
| **现状** | `scripts/seed_data.py` 为 **合成数据**；答辩若强调「真实数据」需与现实现状对齐或升级数据。 |
| **要做事项** | ① 编写或对接合规采集/导出流程（`scripts/scraper.py` 或人工导出 ETL）；② `scripts/clean_data.py`：去重、异常值、字段对齐；③ 更新 `notes` 表结构若需新增字段；④ 重新跑 `compute_baseline.py`；⑤ **文档**：数据来源、时段、样本量、是否含爆款定义（`is_viral` 规则）。 |
| **验收标准** | 仓库内或比赛允许的介质中带 **可复现的清洗说明**；`baseline_stats` 与论文叙事一致（标题长度、标签、时段、封面统计等）。 |
| **主要文件** | `scripts/seed_data.py`（标注 dev-only）、新 `scripts/clean_data.py`、`scripts/compute_baseline.py`、`backend/app/baseline/comparator.py`。 |
| **风险** | 版权与隐私；仅使用可获得授权或公开范围的样本，README 写明边界。 |

### 3.2 Baseline 指标扩展（P1）

| 项目 | 内容 |
|------|------|
| **目标** | 贴近白皮书：标签数量与互动率关系、发布时间 vs 曝光代理指标、粉丝量分层。 |
| **现状** | 已有部分聚合（`compute_baseline.py`）；可与真实数据后再做相关性分析。 |
| **要做事项** | ① 按粉丝分层统计；② 标签数分桶 vs 平均互动；③ 可选：关键词共现/标题模板聚类（演讲素材）。 |
| **验收标准** | Slide 可展示 1–2 张「数据洞察图」来自真实 pipeline。 |

### 3.3 本地数据库与仓库策略

| 项目 | 内容 |
|------|------|
| **目标** | CI/协作者可一键生成 DB，避免提交大二进制。 |
| **现状** | `.gitignore` 忽略 `*.db`；新成员需跑三条脚本。 |
| **要做事项** | README 已部分覆盖；可补充 `Makefile` 或 `justfile`：`make data` = init + seed + compute。 |

## 4. 算法与多模态增强

### 4.1 CLIP 或等价视觉语义（P1）

| 项目 | 内容 |
|------|------|
| **目标** | 白皮书：OpenCV + **CLIP** 视觉语义；用于封面与垂类「视觉风格」叙述。 |
| **现状** | 无 CLIP；`image_analyzer.py` 为传统图像特征 + 简化主色。 |
| **要做事项** | ① 选本地 `open_clip` / API embedding；② 输出标签或向量摘要写入 Agent prompt；③ 若 baseline 无封面向量，先做「单笔记诊断」不强行对比。 |
| **验收标准** | 可视化 Agent 的 reasoning 可引用「语义标签」；体量与部署时间需在 48h 黑客松内可评估。 |
| **主要文件** | `backend/app/analysis/image_analyzer.py`、`requirements.txt`。 |

### 4.2 本地 OCR 备选（P2）

| 项目 | 内容 |
|------|------|
| **目标** | API 不可用或成本受限时仍可 OCR。 |
| **现状** | 未集成 PaddleOCR 等。 |
| **要做事项** | 可选依赖包 + `ocr_processor.py` 分支；Docker/文档说明模型体积。 |

### 4.3 封面「构图」叙述增强（P2）

| 项目 | 内容 |
|------|------|
| **目标** | 答辩用语：不只饱和度，还有构图/主体突出等。 |
| **现状** | 有边缘估计文字占比、简单分块主色。 |
| **要做事项** | 显著性/主体检测简化版或 LLM 直接读图加强描述（与 OCR 共用视觉模型时可合并请求）。 |

## 5. 多 Agent 与可解释性

### 5.1 辩论过程结构化与前端时间线（P1）

| 项目 | 内容 |
|------|------|
| **目标** | 白皮书卖点：Agent 互相质疑；**可展示给用户看**。 |
| **现状** | 后端第二轮辩论存在，但响应体以裁判摘要为主；前端无「回合制」气泡链。 |
| **要做事项** | ① `orchestrator.py` 组装 `debate_timeline: [{ round, agent_name, kind: agree|rebuttal|add, text }]`；② Pydantic `DiagnoseResponse` 扩展字段；③ `AgentDebate.tsx` 时间线 UI；④ 与 `DEBATE_PROMPT` 返回字段对齐（agreements/disagreements/additions）。 |
| **验收标准** | 演示时可展开「辩论时间线」3+ 条可见记录，与后端 JSON 一致。 |
| **主要文件** | `backend/app/agents/orchestrator.py`、`backend/app/models/schemas.py`、`frontend/src/utils/api.ts`、`frontend/src/components/AgentDebate.tsx`。 |

### 5.2 Token 与并行策略优化（P1）

| 项目 | 内容 |
|------|------|
| **目标** | 白皮书：前四 Agent 并行，裁判串行；控制费用与延迟。 |
| **现状** | 已 `asyncio.gather` 第一轮；辩论轮 4 次 LLM + 裁判，总调用次数较高。 |
| **要做事项** | ① 记录各阶段耗时与 token；② 可选：辩论改为 2 Agent 对抗 + 裁判；③ 模型降级路径（`gpt-4o-mini`）仅用于非裁判。 |
| **验收标准** | 单次诊断时间与成本可查（日志或 metrics）；答辩能答「为什么这样设计」。 |

### 5.3 多 LLM 提供商（P2）

| 项目 | 内容 |
|------|------|
| **目标** | `requirements.txt` 含 Anthropic，代码路径统一可切换。 |
| **现状** | `base_agent.py` 以 OpenAI 兼容客户端为主。 |
| **要做事项** | `LLM_PROVIDER` 环境变量 + 工厂函数；文档写明比赛现场主用哪家。 |

## 8. 建议排期与里程碑

以下为 **48h 黑客松风格** 的粗粒度里程碑，可按队员分工微调。

| 阶段 | 时长建议 | 聚焦 |
|------|-----------|------|
| **M0** | 赛前 | 真实数据 pipeline 或合法样本；API Key；路演故事定稿 |
| **M1** | 0–8h | 链接或 OCR 选一做到 **P0 闭环**；免责声明 |
| **M2** | 8–20h | 辩论时间线 + 进度体验；Baseline 文档与数字对齐叙事 |
| **M3** | 20–36h | 一键优化最小版；预录/预跑 fallback；EXPO 设备验证 |
| **M4** | 36–48h | 视频、Slide、笔记、Tag；只做 P0 bugfix |
