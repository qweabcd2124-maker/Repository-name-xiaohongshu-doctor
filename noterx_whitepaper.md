# 薯医 / NoteRx

## 项目一句话介绍

**AI驱动的小红书笔记诊断平台——用数据告诉你，你的笔记为什么没火。**

## 三、核心卖点（对齐评审维度）

### 卖点1：真实痛点，真金白银

小红书上有超过8000万创作者。绝大多数素人博主发了笔记没有流量，但不知道问题出在哪——是封面不行？标题不吸引？标签选错了？发布时间不对？目前没有任何工具能给出系统性的诊断。

**薯医就是这个工具。**

→ 对应评审维度：**商业潜力**

### 卖点2：不是玄学，是数据科学

我们不是让AI随便给建议。我们针对2-3个垂类（美食/穿搭/科技），提前采集数千条真实小红书笔记数据，建立**量化评价基线（Baseline）**：

- 该垂类爆款笔记的标题平均字数、关键词分布
- 封面构图的色彩饱和度、文字占比、人脸出现率
- 标签使用策略与互动率的相关性
- 发布时间与曝光量的关系
- 博主身份粉丝量

用户的笔记不是和"AI的感觉"对比，是和**真实数据**对比。这个数据体系就是我们的护城河。

→ 对应评审维度：**技术难度**

### 卖点3：多Agent辩论，不是一言堂

单个AI给建议容易片面。薯医采用**多Agent架构**：

- **内容分析师Agent**：分析文案结构、信息密度、可读性
- **视觉诊断Agent**：分析封面构图、配色、视觉吸引力
- **增长策略Agent**：分析标签、发布时间、互动策略
- **用户模拟Agent**：模拟目标受众的第一反应（融合AI评论区概念）
- **综合裁判Agent**：汇总各Agent观点，处理分歧，给出最终诊断报告

Agent之间会互相质疑——比如视觉Agent说封面很好，但用户模拟Agent说目标用户其实不喜欢这种风格——这种辩论过程本身就是产品亮点，可以展示给用户看。

→ 对应评审维度：**创新性 + 技术难度**

### 卖点4：诊断报告天然是小红书内容

薯医生成的诊断报告是一张精心设计的**诊断卡片**——包含评分雷达图、问题标注、优化建议——用户拿到这张卡就会想发到小红书上："我的笔记被AI诊断了，原来问题出在这里！"

这和PlanCoach的传播逻辑一模一样：产品输出物本身就是社交货币。

→ 对应评审维度：**商业潜力（增长飞轮）**

### 卖点5：我们自己就是用户

队员陈宇夏是自媒体博主，真实运营小红书账号。这不是一群程序员在想象用户需求，这是创作者在解决自己的问题。

→ 对应评审维度：**现场讲述效果**

## 四、产品功能拆解

### 核心流程

```
用户上传笔记（截图/文字/链接）
         ↓
    多模态内容解析
  （OCR + 图像分析 + NLP）
         ↓
   与垂类Baseline数据对比
         ↓
    多Agent并行诊断
  （内容/视觉/增长/用户模拟）
         ↓
    Agent辩论 & 综合裁判
         ↓
   生成诊断报告 + 优化建议
         ↓
   一键生成优化版笔记（可选）
         ↓
   导出诊断卡片（可分享）
```

### 功能模块

| 模块 | 说明 | 优先级 |
|------|------|--------|
| 笔记上传 | 支持截图、粘贴文字、输入链接 | P0 |
| 封面诊断 | 构图分析、色彩评分、文字占比、人脸检测 | P0 |
| 标题诊断 | 关键词吸引力、长度、情绪词、与垂类爆款对比 | P0 |
| 内容诊断 | 结构完整性、信息密度、可读性评分 | P0 |
| 标签诊断 | 标签数量、热度匹配、长尾策略 | P1 |
| 用户预测 | AI模拟评论区反应（融合AI评论区） | P1 |
| Agent辩论可视化 | 展示多Agent讨论过程 | P1 |
| 一键优化 | AI重写标题/重排内容/推荐封面方向 | P2 |
| 诊断卡片导出 | 生成可分享的诊断结果图 | P0 |

## 八、数据科学体系——从「拍脑袋」到「控变量实验」

### 问题

市面上所有「笔记诊断」工具的建议都是 AI 凭感觉给的。用户问「我标题该怎么取？」，AI 说「建议简短有力」——但多短才叫短？什么叫有力？没有数据支撑的建议不可执行。

### 我们的做法

我们为每个垂类采集了 **500+ 条真实小红书笔记**的完整数据（标题、正文、标签、封面、互动数据），然后做了一般创作者工具不会做的事：**控制变量的交叉分析**。

#### 8.1 多维度控变量分析

| 分析维度 | 控制变量 | 观测变量 | 输出结论示例 |
|----------|----------|----------|------------|
| 标题长度 | 同垂类、同粉丝量级、相似封面质量 | 浏览量、点赞率 | 美食垂类 16-20 字标题浏览量中位数比 8-12 字高 47% |
| 标题类型 | 同垂类、同内容质量 | 互动率 | 数字型标题（「5 步搞定」）比陈述型标题互动率高 32% |
| 封面风格 | 同垂类、同标题吸引力 | 点击率 | 人物出镜封面比纯产品图点击率高 28%（穿搭垂类） |
| 标签策略 | 同内容质量、同发布时间 | 曝光量 | 5-8 个标签（含 2-3 个热门）比 1-3 个标签曝光量高 60% |
| 发布时间 | 同内容、同账号 | 24h 互动量 | 美食笔记 17:00-19:00 发布比凌晨发布互动量高 3 倍 |

#### 8.2 双层评价系统

我们不是只跑一遍统计，而是建了**两套互补的评价体系**：

**第一层：量化基线数据库（Quantitative Baseline）**

```
原始数据（500+条/垂类）
     ↓
多维特征提取（标题长度/关键词/封面色彩/标签/时段...）
     ↓
分组聚类（按粉丝量级、按内容类型、按互动表现）
     ↓
控变量回归分析
     ↓
输出：每个垂类的量化评价标准
  - 标题最佳长度区间
  - 封面最佳饱和度/文字占比
  - 标签最优数量和组合策略
  - 最佳发布时间段
```

这套系统回答的是「好的笔记长什么样」——用数字说话。

**第二层：案例匹配引擎（Case-Matching Engine）**

当用户提交笔记后，系统不只是跟统计均值对比，还会：

1. 在语料库中找到与用户笔记**最相似的 5-10 篇笔记**（同垂类、相似主题、相似标签）
2. 在这些相似笔记中，区分爆款和普通笔记
3. 分析差异：**同样写蛋炒饭，为什么 A 爆了 B 没爆？**
4. 将差异点作为具体建议返回给用户

这套系统回答的是「和你类似的笔记里，成功的那些做对了什么」——用案例说话。

两层系统的输出会同时送入多 Agent 诊断流程，Agent 拿到的不是模糊的指令，而是有数据支撑的分析材料。这就是为什么薯医的建议比其他 AI 工具**具体得多**。

## 九、自进化系统——越用越聪明

### 核心理念

大多数 AI 工具是静态的：第一天能做什么，第一百天还是做什么。薯医不一样——我们设计了一套**数据飞轮**，让系统在使用过程中持续进化。

### 9.1 数据回流架构

```
用户提交笔记
     ↓
AI 诊断 → 生成报告
     ↓                    ↓
用户查看报告         用户反馈（可选）
     ↓                    ↓
  ┌──────────────────────────────────┐
  │         反馈收集层               │
  │                                 │
  │  👍/👎 单条建议有用/没用         │
  │  ⭐ 报告整体评分（1-5星）        │
  │  📸 用户授权存档截图（可选）      │
  │  📊 用户后续数据（改了之后火没火）│
  └────────────┬─────────────────────┘
               ↓
  ┌──────────────────────────────────┐
  │       自进化引擎                  │
  │                                  │
  │  1. 扩充语料库（新案例不断积累）   │
  │  2. 修正基线（动态更新统计指标）   │
  │  3. 优化 Prompt（根据反馈调权重） │
  │  4. 训练判断力（A/B测试建议效果） │
  └──────────────────────────────────┘
```

### 9.2 四种反馈信号

| 反馈类型 | 获取方式 | 数据价值 | 隐私处理 |
|----------|----------|----------|----------|
| **建议评价** | 每条建议旁的点赞/点踩 | 直接信号：哪些建议用户觉得有用 | 匿名聚合，不关联用户身份 |
| **报告评分** | 报告底部 1-5 星评价 | 整体满意度，用于评估 Agent 质量 | 匿名聚合 |
| **授权存档** | 用户主动勾选「允许用于训练」 | 扩充真实笔记语料库，尤其是截图中的封面数据 | 明确知情同意，可随时撤回 |
| **效果追踪** | 用户再次诊断同一笔记 | 闭环验证：改了之后分数变了吗？ | 仅比对前后差异，不关联账号 |

### 9.3 进化路径

**短期（赛期内）**：收集评价数据，人工复盘 Agent 诊断质量
**中期（赛后 1-3 月）**：用累积的反馈数据自动调整 Agent prompt 权重（哪些维度用户最关心、哪些建议采纳率最高）
**长期**：积累足够多的「建议→效果」闭环数据后，可以做到**个性化诊断**——根据博主的历史数据和粉丝画像，给出定制化的建议

### 9.4 为什么这很重要

这不是功能堆砌，这是**商业模式的核心壁垒**。

- 第一个用户用薯医，体验和第 10000 个用户是一样的——这是普通 AI 工具的天花板
- 而薯医的第 10000 个用户，会享受到前 9999 个用户的数据积累带来的更精准诊断——这就是数据飞轮

竞争对手可以抄我们的 UI，抄我们的 Agent 架构，但抄不走我们积累的数据和用户反馈。**时间站在我们这边。**
